### Why ProjectReference Instead of PackageReference?

**Problem**: Code coverage tools cannot instrument code from NuGet packages.

**Solution**: The test project uses `<ProjectReference>` instead of `<PackageReference>` to enable coverage instrumentation.

**Native Library Handling**: A custom MSBuild target copies native CFITSIO libraries from the source project to the test output:

```xml
<Target Name="CopyNativeAssets" AfterTargets="Build">
  <ItemGroup>
    <NativeLibs Include="..\HVO.Astronomy.CFITSIO\runtimes\**\*.*" />
  </ItemGroup>
  <Copy SourceFiles="@(NativeLibs)" 
        DestinationFiles="@(NativeLibs->'$(OutDir)runtimes\%(RecursiveDir)%(Filename)%(Extension)')" 
        SkipUnchangedFiles="true" />
</Target>
```

This ensures:
- ✅ Coverage instrumentation works (ProjectReference)
- ✅ Native libraries load correctly (custom copy target)
- ✅ Tests run successfully on all platforms

### Coverage Limitations
The HVO.Astronomy.CFITSIO library consists primarily of P/Invoke declarations (LibraryImport) which are:
- Source-generated by the compiler
- Marked with `[GeneratedCodeAttribute]` and `[CompilerGeneratedAttribute]`
- Automatically excluded from code coverage by standard coverage tools

The managed wrapper classes (`FitsFile`, `FitsImage`, `FitsHeaderBuilder`, `WcsHeaderBuilder`, etc.) contain the actual business logic:
- ~1,579 lines of C# code
- Result<T> pattern for error handling
- Type-safe wrappers around native CFITSIO functions
- Helper methods for common FITS operations

### Coverage Tools Note
Standard .NET coverage tools (coverlet, dotCover) exclude:
- P/Invoke extern methods
- Source-generated code  
- Compiler-generated code (auto-properties, lambda closures, etc.)

Therefore, coverage metrics for this library focus on:
- ✅ **Functional validation** via integration tests
- ✅ **Entry point validation** via native library tests  
- ✅ **Cross-platform compatibility** verification
- ❌ **Line coverage metrics** (not meaningful for P/Invoke-heavy libraries)

## Known Limitations

### 64-bit Integer Keywords
- **Issue**: CFITSIO doesn't have native 64-bit integer keyword functions (`ffpkyjj`/`ffukyjj` don't exist)
- **Solution**: String-based storage with automatic parsing
- **Impact**: Int64 values stored as strings, parsed on read
- **Test Coverage**: ✅ Validated in `HeaderBuilder_Writes_Typed_Keywords`

### USHORT_IMG BITPIX Normalization
- **Behavior**: CFITSIO normalizes USHORT_IMG (20) to BITPIX=16 with BZERO=32768
- **Impact**: Tests expect SHORT_IMG (16), not USHORT_IMG (20)
- **Test Coverage**: ✅ Validated in `Create_Write_Read_U16_Image_Succeeds`

### Double Precision Formatting
- **Issue**: CFITSIO `decimals=-1` may not preserve full precision for all values
- **Solution**: Use explicit `decimals=15` for WCS coordinates and high-precision values
- **Test Coverage**: ✅ Validated in `WcsBuilder_Writes_TAN_With_CD_Matrix`

## CI/CD Integration

To run these tests in CI:

```bash
# Run all tests
dotnet test src/HVO.Astronomy.CFITSIO.Tests/HVO.Astronomy.CFITSIO.Tests.csproj

# Run only native library validation tests
dotnet test --filter "FullyQualifiedName~NativeLibraryTests"

# Run with coverage (limited usefulness for P/Invoke libraries)
dotnet test --collect:"XPlat Code Coverage" --settings src/HVO.Astronomy.CFITSIO.Tests/cfitsio.coverage.runsettings
```

The native library tests are especially valuable in CI to catch:
- Missing native library files in runtime folders
- Platform-specific entry point issues
- ABI compatibility problems
- Packaging/deployment issues
